# v2 재구축 계획

작성일: 2026-07-02.

## 목표

현재 저장소의 v1을 직접 수리하지 않고, 이 폴더 안에서 `v2`를 새 제품처럼 재구축한다. v1은 검증된 도메인 지식과 UI 참고 자료로만 사용하고, 의류 데이터 스키마·수집 파이프라인·추천 로직·프로젝트 구조는 새로 정리한다.

## 핵심 가정

- `v2/`는 새 앱과 문서의 기준 폴더다.
- `v2_md/`는 기존 문서 보관 폴더로 보고, 새 작업의 기준은 `v2/`로 통일한다.
- v2 UI 구성은 v1 원본 화면 흐름과 룩앤필을 계승한다.
- 기존 v2 골든 패스 대시보드는 최종 UI가 아니라 추천 엔진과 프리셋 검증용 smoke prototype으로만 보존한다.
- 골든 패스는 외부 네트워크 없이 돌아야 한다.
- 골든 패스는 원본 앱처럼 퍼스널컬러 측정, 결과, 옷장 구성, 추천, 저장 코디, 데일리룩 저장 흐름을 포함한다.
- 사진 등록과 URL 등록은 1차 시연 경로 밖의 부가 기능이지만, 원본 UI의 진입점은 유지한다.
- 퍼스널컬러 시즌과 착용 계절은 타입·문구·로직에서 분리한다.

## 단계

1. P0 문서·계약 정리.
   - `v2/docs/domain-logic-inventory.md`에 v1에서 승계할 로직 정보를 정리한다.
   - `v2/docs/v2-schema-contract.md`에 새 의류·퍼컬·추천 스키마를 고정한다.
   - `v2/checklist.md`, `v2/context-notes.md`를 날짜별 누적 문서로 유지한다.

2. P0 스캐폴드.
   - React 19 + TypeScript + Vite 앱을 `v2` 루트 패키지로 구성한다.
   - v1 원본 UI 소스와 공용 UI 컴포넌트를 `v2/` 안으로 가져와 독립 실행 기준을 만든다.
   - v2 prototype 대시보드는 `v2/src/prototype` 아래로 분리해 최종 앱 진입점에서 제외한다.
   - FastAPI 서버는 `v2/server`에 새로 둔다.
   - 데이터 파이프라인은 `v2/pipeline`, 설정은 `v2/config`에 둔다.

3. P1 골든 패스.
   - 원본 홈, 퍼컬 측정, 설문, 결과, 옷장, 추천, 저장 코디, 데일리룩 화면 흐름을 유지한다.
   - 카탈로그 기반 프리셋 옷장을 먼저 만들되 데이터 원본과 메타데이터를 `v2/catalog`에 모은다.
   - 겹치지 않는 추천 3개가 항상 성립하도록 도메인 테스트를 둔다.
   - 추천 결과를 데일리룩으로 저장하고 이미지로 내보내는 원본 UI 경로를 유지한다.

4. P2 부가 수집 경로.
   - URL 수집과 사진 등록을 같은 의류 분석 스키마로 연결한다.
   - 자동 인식은 초안 작성기로만 쓰고, 사용자 확정값을 최종 권한으로 둔다.

5. P3 논문 트랙.
   - v1 RF, 임베딩 분류 헤드, VLM 제로샷을 동일 라벨 516장으로 비교한다.
   - 서비스 구현과 논문 실험을 분리해 일정 리스크를 줄인다.

## 1차 완료 기준

- `npm test` 통과.
- `npm run build` 통과.
- 프리셋 옷장에서 겹치지 않는 추천 3개가 나온다.
- 추천 근거와 축별 점수가 화면에 표시된다.
- 데일리룩 이미지 저장이 된다.
- 골든 패스는 네트워크 없이 완주된다.

## 2026-07-03 추가 계획

### 색상 시각화와 로컬 테스트 데이터 보강

사용자가 요청한 색상 파트는 기존 정적 HTML을 단순 팔레트 표가 아니라 계산 모델 설명서로 올리는 작업이다. `HEX -> RGB -> CIELAB -> LCh` 변환, CIEDE2000 색차, Delta E 점수 매핑, 시즌별 LCh 영역, 회피색 결정 경계, 색채 조화 각도, 새 스펙트럼 표현을 한 문서 안에서 확인할 수 있어야 한다.

사진을 찍지 않아도 앱 추천 흐름을 시험할 수 있도록 가상의 여름뮤트, 즉 `soft-summer` 결과를 기본 데모 사용자 데이터로 저장한다. 실제 사용자가 이미 퍼컬 결과를 저장한 경우에는 덮어쓰지 않고, 저장값이 없을 때만 초기값으로 쓴다.

옷장 초기화 문제는 서버 DB 문제가 아니라 브라우저 `localStorage` 복원 단계의 카탈로그 ID 필터 문제로 본다. v2 카탈로그 ID는 `catalog-` 접두사만 쓰지 않으므로, 저장된 카탈로그 의류가 현재 카탈로그에 존재하면 재시작 후에도 보존해야 한다.

### 검증 기준

- 색상 HTML 정적 테스트가 Lab, LCh, CIEDE2000, 시즌 영역, 여름뮤트 데모 설명을 확인한다.
- 옷장 복원 테스트가 `upper_shirt_001`처럼 `catalog-`로 시작하지 않는 v2 카탈로그 ID를 유지한다.
- 퍼컬 데모 데이터 테스트가 저장값이 없을 때 `soft-summer` 결과를 제공하고, 기존 저장값을 덮어쓰지 않음을 확인한다.
- `npm test`와 `npm run build`를 통과한다.
- 저장 계층 DB 전환 준비.

PostgreSQL은 지금 붙이지 않는다. 대신 현재 브라우저 저장 구현을 도메인 저장소 어댑터로 감싸서 훅과 화면이 `localStorage` 키를 직접 알지 않게 만든다. 이번 범위는 저장 방식 변경이 아니라 저장 경계 정리다.

목표는 퍼컬 결과와 이력, 옷장과 의류 목록, 저장 코디와 데일리룩 저장을 `services` 계층의 어댑터로 모으는 것이다. 기존 `imageStore.ts`의 IndexedDB 이미지 오프로드 구조는 유지한다.

검증 기준은 새 저장소 어댑터 테스트, 기존 옷장 복원 테스트, 여름뮤트 데모 사용자 테스트, 저장 코디 흐름 테스트, `npm test`, `npm run build`다.

### 옷장 색상 분석 팝업 레이어

옷장 상세 화면에서 의류 색상 정보를 누르면 팝업 레이어로 `HEX -> RGB -> CIELAB -> LCh`, CIEDE2000, 시즌 스펙트럼 영역, 색채이론 요약을 확인할 수 있게 한다. 사이드바와 현재 전체 UI 구조는 유지한다.

이번 범위는 URL 입력이나 인터넷 사진 분석 수집 기능 구현이 아니다. 다만 나중에 주소/사진 분석 완료 화면에서도 같은 색상 분석 패널을 재사용할 수 있도록 계산 계층과 표시 컴포넌트를 옷장 상세 내부 전용 로직으로 숨기지 않는다.

검증 기준은 색상 분석 서비스 테스트, 팝업 렌더링 테스트, 옷장 상세 색상 버튼 계약 테스트, `npm test`, `npm run build`다.

### URL 수집 1단계 — FastAPI 프록시와 SSRF 가드

URL 수집 기능은 브라우저 CORS와 외부 이미지 canvas 제한 때문에 서버 프록시가 선행되어야 한다. `v2/server`에 FastAPI 앱을 새로 두고 `POST /api/ingest/url` 단일 엔드포인트를 만든다.

이번 1단계는 상품 분석 완성이나 사이트별 스크래퍼가 아니다. 목표는 http/https만 허용하고, 사설 IP·localhost·link-local·reserved 주소를 막고, 리다이렉트마다 같은 검사를 반복하며, 응답 크기를 제한하고, Content-Type 기준으로 이미지와 HTML을 분기하는 안전한 수집 골격이다.

테스트는 실네트워크 없이 가짜 fetcher와 resolver로 실행한다. 검증 기준은 URL 스킴 차단, 사설 IP 차단, 안전한 리다이렉트 허용, 사설 IP 리다이렉트 차단, 크기 제한, 이미지/HTML 응답 분기, FastAPI 라우트 계약, `python -m unittest`, `npm test`, `npm run build`다.

### URL 입력 UI 1차 연결

서버 프록시만으로는 사용자가 주소를 넣을 수 없으므로 기존 `나만의 옷 추가` 화면 안에 `URL로 가져오기` 입력 경로를 추가한다. 새 페이지를 만들지 않고 사진 업로드와 URL 입력을 같은 수동 등록 화면의 입력 방식으로 나눈다.

이번 범위는 URL 분석 완료 초안 저장까지가 아니다. 사용자가 쇼핑몰 주소나 이미지 주소를 넣고 `POST /api/ingest/url`을 호출해 서버가 안전하게 확인한 결과를 화면에서 볼 수 있게 한다. 이미지/HTML 결과가 오면 다음 단계 안내를 보여주고, 실제 대표 이미지 파싱·누끼·색 추출·초안 생성은 다음 단계로 남긴다.

검증 기준은 URL ingest API 호출 테스트, 수동 등록 화면 URL 입력 계약 테스트, `npm test`, `npm run build`다.

## 2026-07-03 URL 상품 페이지 대표 이미지 파싱

이번 단계는 이미 동작하는 `POST /api/ingest/url` 프록시 위에 상품 페이지 파싱을 얹는다. 범위는 HTML 응답에서 대표 이미지와 상품명 초안을 찾고, 수동 등록 화면의 URL 결과 패널에 분석 가능한 대표 이미지 상태를 보여주는 데까지다.

파싱 순서는 FRD 원칙대로 `og:image`, JSON-LD `schema.org/Product`, 이미지 태그 휴리스틱 순서로 고정한다. 사이트별 스크래퍼는 만들지 않고, 실패는 정상 폴백으로 처리해 이미지 주소 직접 입력이나 사진 업로드로 이어지게 한다.

검증 기준은 서버 단위 테스트에서 세 가지 파싱 경로와 실패 폴백을 모두 통과하고, 프론트 테스트에서 새 응답 필드와 결과 화면 문구를 확인한 뒤 전체 `npm test`와 `npm run build`를 통과하는 것이다.

## 2026-07-03 URL 대표 이미지 초안 연결

이번 단계는 대표 이미지 찾기 완료 상태에서 실제 등록 초안까지 연결한다. 브라우저는 외부 쇼핑몰 CDN 이미지를 CORS 제한 때문에 직접 내려받아 분석할 수 없으므로, 서버에 같은 SSRF 가드를 쓰는 이미지 프록시 `POST /api/ingest/image`를 추가한다.

프론트는 프록시로 받은 이미지를 사진 업로드와 완전히 같은 자동 분석 경로(정밀 누끼, 색 추출, 분류)에 태운다. 별도 저장 경로를 만들지 않고, 기존 옷장에 저장 버튼이 사용자 확정 지점이 된다. 저장되는 아이템은 sourceType 'url'과 출처 주소 sourceRef를 기록한다.

검증 기준은 서버 이미지 프록시 테스트(이미지 반환, 비이미지 차단, 사설망 차단), 프론트 API·화면 계약 테스트, 전체 `npm test`와 `npm run build`, 그리고 무신사 실제 URL로 초안 생성부터 옷장 저장까지 브라우저 확인이다.

## 2026-07-08 퍼컬 진단 백지 화면 복구

3100번 v2 로컬 앱에서 홈의 `측정 시작`을 누르면 `PhotoAnalyzer`가 열리기 전에 React 루트가 비는 문제를 복구한다. 재현 로그는 `@base-ui/react/button`의 `Invalid hook call`과 `Cannot read properties of null (reading 'useRef')`이다.

원인 가설은 v2 앱이 `react@19.2.7`을 쓰는 반면, v2에 설치되지 않은 `@base-ui/react`가 상위 루트 `node_modules`에서 해석되어 `react@19.2.5`와 섞이는 것이다. 이번 수정은 퍼컬 진단 화면에서 필요한 단순 버튼과 진행바를 v2 내부 네이티브 React 컴포넌트로 바꾸어 상위 Base UI 의존성을 타지 않게 하는 데 한정한다.

성공 기준은 다음과 같다.

- v2 UI 원시 컴포넌트가 `@base-ui/react`를 import하지 않는 회귀 테스트가 먼저 실패한 뒤 통과한다.
- `npm test`와 `npm run build`가 통과한다.
- 브라우저에서 `http://localhost:3100` 홈의 `측정 시작`을 눌러도 루트가 비지 않고 사진 분석 모듈이 보인다.
- 브라우저 콘솔에 `Invalid hook call` 오류가 다시 나오지 않는다.


## 2026-07-10 ColorFit UI 전면 이식

`colorfit-ui-v6.zip`은 화면을 교체하는 정적 결과물이 아니라, 실제 `v2` 화면에 적용할 승인된 시각 기준으로 사용한다. 현재 기능과 상태 소유권은 유지하고 각 React 컴포넌트의 정보 구조와 CSS를 V6 방향에 맞춘다.

### 구현 원칙

- 홈은 직전 1단계 변경본보다 원본 앱의 네 가지 핵심 진입 동선인 퍼컬 진단, 옷장 만들기, 옷 추가, 데일리룩을 우선한다.
- 일반 화면은 불투명도가 높은 차분한 표면을 사용하고, 퍼컬 촬영·설문·결과와 색상 분석 팝업에만 리퀴드 글라스를 강하게 사용한다.
- 퍼컬 결과는 실제 판정 시즌을 기준으로 24색 팔레트, 4축 프로필, 관찰 색상, 스펙트럼 위치, 옷장 활용색을 보여준다. 12계절 비교는 표시용이며 저장된 판정값을 바꾸지 않는다.
- 옷장 상세의 준비도·건강도 평가는 제거하고 실제 카테고리 수와 자주 쓰는 색을 보여준다.
- 옷 추가는 이미지 분석 결과를 먼저 보여주고, 사용자가 `정보 수정`을 열었을 때만 상세 입력 시트를 표시한다. 모바일 시트는 아래에서 열리고 스와이프로 닫을 수 있어야 한다.
- 옷 추가의 추출 색상과 옷장 의류 대표색은 같은 `ColorInsightModal`을 사용한다.
- 추천은 조건을 짧게 요약하고 1순위 룩의 이미지, 점수, 근거, 저장 행동을 가장 크게 보여준다. 내부 점수 진단용 표현은 기본 화면에서 제거한다.
- 룩 보관함은 폴더와 즐겨찾기 분류를 실제 저장 데이터에 반영한다.
- 데일리룩 편집은 PC에서 자산·캔버스·레이어 3열, 모바일에서 캔버스 우선 + 하단 편집 패널로 구성한다. 드래그 업데이트는 프레임 단위로 제한한다.

### 검증 기준
