# ColorFit 독립 서비스 기능 요구사항 정의서

작성일: 2026-07-16.

문서 상태: 검토 요청 초안.

대상 릴리스: ColorFit 독립 공개 서비스 1.0.

기준 소스: `colorfit_ax.zip`의 Git HEAD `18f0ffa`와 현재 저장소 `v2`.

## 1. 문서 목적

이 문서는 ColorFit v2를 부모 저장소에 의존하지 않는 하나의 공개 서비스로 완성하기 위한 제품 요구사항을 정의한다. 현재 ColorFit UI와 검증된 프론트 기능은 유지하고, 독립화 과정에서 누락된 ML 이미지 처리, 잘못된 초기 데모 상태, 배포 경계와 장애 대체 경로를 제품 계약으로 고정한다.

이 문서는 구현 상세 설계가 아니다. 사용자가 무엇을 할 수 있어야 하는지, 시스템이 어떤 상태와 실패를 처리해야 하는지, 어떤 조건을 만족해야 출시할 수 있는지를 정의한다.

## 2. 배경과 문제 정의

### 2.1 확인된 기준선

`colorfit_ax.zip`은 부모 저장소 없이 다음 검증을 통과했다.

- `npm ci`가 성공했다.
- 프로덕션 빌드가 성공했다.
- 27개 테스트 파일과 142개 테스트가 통과했다.
- 얼굴 촬영, MediaPipe 얼굴 검출, 사진 색상 분석, 설문, 퍼컬 융합, 날씨, 옷장, 추천, 저장 로직이 포함되어 있다.

따라서 현재 문제를 UI만 남은 프로토타입으로 정의하지 않는다.

### 2.2 확인된 핵심 결함

1. 독립 저장소에 실제 ML 누끼 구현인 `server/background_remove_api.py`가 없다.
2. `server/ml_bridge.py`는 프로젝트 밖의 부모 경로에서 위 파일을 찾고 실패를 숨긴다.
3. Vercel 엔트리는 `/api/background/remove`와 `/api/clothing/extract`를 HTTP 503으로 고정한다.
4. 신규 사용자에게 실제 진단 없이 소프트 서머 데모 결과가 자동 적용된다.
5. 배포 코드, 카탈로그 원본, 공개 자산, 빌드 산출물과 작업 아카이브의 경계가 불명확하다.
6. 현재 테스트는 프론트 회귀를 방어하지만 독립 ML 서비스의 실제 성공을 검증하지 않는다.

### 2.3 해결할 제품 문제

사용자는 공개 URL에 접속해 실제 얼굴 촬영과 설문으로 퍼스널컬러를 진단하고, 자신의 옷을 등록해 날씨와 퍼스널컬러에 맞는 코디를 추천받을 수 있어야 한다. 로컬 개발 환경이나 v1 부모 폴더가 없어도 같은 핵심 흐름이 동작해야 한다.

## 3. 제품 비전과 원칙

ColorFit은 사용자의 얼굴 사진과 설문, 실제 보유 의류, 현재 날씨를 하나의 로컬 우선 흐름으로 연결해 설명 가능한 코디를 추천하는 개인 스타일 도구다.

제품은 다음 원칙을 따른다.

- 실제 데이터가 없는 상태를 완성된 결과처럼 꾸미지 않는다.
- 사진 분석과 추천 근거를 사용자에게 설명한다.
- 자동 분석 결과보다 사용자의 확인과 수정이 우선한다.
- ML 기능 장애가 전체 의류 등록을 막지 않는다.
- 브라우저에 저장된 개인 데이터는 사용자의 통제 아래 둔다.
- 독립 저장소 하나만으로 프론트 빌드와 배포가 가능해야 한다.

## 4. 목표

### G-01 실제 퍼스널컬러 진단

사용자가 카메라 또는 사진과 설문으로 실제 12계절 결과를 얻을 수 있어야 한다.

### G-02 실제 디지털 옷장

사용자가 카탈로그, 사진 업로드, 상품 URL로 옷을 등록하고 수정·삭제·분류할 수 있어야 한다.

### G-03 설명 가능한 추천

퍼스널컬러, 날씨, 색상 조화, 착용 안정성의 네 축으로 겹치지 않는 추천 조합과 근거를 제공해야 한다.

### G-04 독립 공개 서비스

부모 저장소나 로컬 전용 파일 없이 Vercel 공개 URL과 독립 ML API로 핵심 기능을 제공해야 한다.

### G-05 장애에 강한 흐름

날씨, MediaPipe CDN, URL 수집, ML 이미지 처리 중 일부가 실패해도 가능한 수동 대체 경로를 제공해야 한다.

## 5. 비목표

독립 서비스 1.0에서는 다음 항목을 구현하지 않는다.

- 사용자 계정, 소셜 로그인과 서버 데이터 동기화.
- 여러 기기 간 옷장 자동 동기화.
- 결제, 구독, 쇼핑몰 주문과 제휴 구매.
- 소셜 피드와 타인 옷장 공유.
- 관리자 CMS와 ML 모델 재학습 파이프라인.
- 생성형 AI 의류 이미지 제작.
- 의료 수준의 색채 진단 보증.
- 현재 ColorFit UI의 전면 재설계.

## 6. 주요 사용자

- 처음 진단하는 사용자. 자신의 퍼스널컬러를 모르며 휴대전화로 진단하려는 사용자다.
- 옷장을 등록하는 사용자. 사진 또는 상품 URL로 보유 의류를 추가하고 자동 분석을 수정하려는 사용자다.
- 오늘의 코디가 필요한 사용자. 현재 날씨와 실제 보유 의류에 맞는 조합을 추천받으려는 사용자다.
- 결과를 저장하는 사용자. 추천 조합을 편집하고 이미지로 보관하려는 사용자다.

## 7. 핵심 사용자 여정

### J-01 첫 진단

1. 사용자가 공개 ColorFit URL에 접속한다.
2. 앱은 진단 결과가 없는 신규 상태를 표시한다.
3. 사용자가 카메라 권한을 허용하거나 사진을 선택한다.
4. 앱이 얼굴 검출과 사진 품질을 안내한다.
5. 사용자가 촬영 결과를 확인하고 설문을 완료한다.
6. 앱이 사진과 설문을 융합해 Top1, Top2, 신뢰도와 근거를 표시한다.
7. 사용자가 결과를 저장한다.

### J-02 사진으로 옷 등록

1. 사용자가 옷장을 선택하고 사진을 촬영하거나 업로드한다.
2. 원본 사진이 즉시 미리보기로 표시된다.
3. 독립 ML 서비스가 배경 제거, 분류와 대표색 초안을 반환한다.
4. 사용자가 초안을 확인하고 필요한 값을 수정한다.
5. 확정값과 원본·누끼 이미지가 옷장에 저장된다.

### J-03 상품 URL로 옷 등록

1. 사용자가 상품 페이지 또는 이미지 URL을 입력한다.
2. 서버가 SSRF 검사를 통과한 URL만 요청한다.
3. 서버가 상품명과 대표 이미지를 추출한다.
4. 사용자가 이미지를 등록 초안으로 채택한다.
5. 이후 사진 등록과 동일한 분석·확정 흐름을 사용한다.

### J-04 오늘의 추천

1. 앱이 현재 위치 날씨를 요청한다.
2. 실패 시 서울 기준 또는 수동 기온을 제공한다.
3. 사용자가 추천 목적과 대상 옷장을 선택한다.
4. 앱이 겹치지 않는 추천 조합 최대 3개를 만든다.
5. 각 추천에 네 축 점수와 자연어 근거를 표시한다.
6. 사용자가 추천을 저장하거나 데일리룩 편집기로 이동한다.

## 8. 기능 요구사항

### FR-001 신규 사용자 초기 상태

- 저장된 진단 결과가 없는 사용자는 미진단 상태로 시작해야 한다.
- 일반 실행에서 데모 소프트 서머 결과를 자동 적용해서는 안 된다.
- 데모 데이터는 별도 플래그 또는 별도 데모 배포에서만 사용할 수 있다.
- 진단 결과가 없을 때 추천 화면은 진단 필요 상태를 표시해야 한다.

인수 기준.

- 브라우저 저장소를 비운 뒤 첫 접속하면 시즌명, 신뢰도와 데모 이력이 표시되지 않는다.
- 진단 완료 전에는 실제 결과처럼 보이는 추천을 생성하지 않는다.

### FR-010 카메라와 사진 입력

- 지원 브라우저에서 전면 카메라를 요청해야 한다.
- 카메라 미지원, 권한 거부와 장치 오류를 구분해 안내해야 한다.
- 카메라를 사용할 수 없으면 사진 파일 업로드를 제공해야 한다.
- 얼굴 위치, 조명, 안경과 강한 색조 화장에 대한 최소 안내를 제공해야 한다.
- 사용자가 확정하기 전 촬영 이미지를 진단 결과로 저장하지 않아야 한다.

### FR-011 얼굴 검출과 사진 품질

- MediaPipe Face Landmarker로 얼굴과 기준점을 검출해야 한다.
- GPU 초기화 실패 시 CPU 대체를 시도해야 한다.
- 얼굴 미검출, 다중 얼굴, 너무 작은 얼굴과 잘못된 노출을 안내해야 한다.
- 피부, 머리, 눈, 입술의 분석 영역과 조명 보정 품질을 계산해야 한다.
- 품질이 낮으면 사진 비중을 낮추고 재촬영을 권해야 한다.

### FR-012 설문과 퍼컬 융합

- 설문은 온도, 명도, 청탁, 대비의 네 축을 계산해야 한다.
- 사용자는 이전 문항으로 돌아가 응답을 수정할 수 있어야 한다.
- 누락 문항이 있으면 결과 계산을 시작하지 않아야 한다.
- 결과는 12계절 중 Top1과 Top2, 신뢰도, 판정 방식과 사진·설문 가중치를 반환해야 한다.
- 경계 판정, 추천 팔레트, 관찰 색상, 4축 프로필과 설명을 표시해야 한다.
- 12계절 비교 UI는 저장된 진단 결과를 바꾸지 않아야 한다.
- 사용자는 재진단하고 과거 진단을 다시 적용할 수 있어야 한다.

### FR-020 날씨

- 브라우저 위치 권한으로 현재 날씨를 조회해야 한다.
- 현재 기온, 체감온도, 강수, 날씨 상태와 기온 구간을 제공해야 한다.
- 위치 권한 거부와 API 실패를 정상 시나리오로 처리해야 한다.
- 실패 시 서울 기준임을 표시하고 수동 기온 구간 변경을 허용해야 한다.
- 수동 변경값을 해당 추천 세션에서 자동 날씨가 덮어쓰지 않아야 한다.

### FR-030 옷장 관리

- 사용자는 옷장을 생성하고 이름을 변경하고 삭제할 수 있어야 한다.
- 옷장 삭제 전 포함 의류 수와 영향을 확인해야 한다.
- 의류는 수정, 삭제, 카테고리 필터와 검색이 가능해야 한다.
- 추천 제외 상태인 의류는 추천 후보에 포함하지 않아야 한다.

### FR-031 카탈로그 등록

- 준비된 카탈로그 의류를 기존 또는 새 옷장에 추가할 수 있어야 한다.
- 같은 카탈로그 항목을 같은 옷장에 중복 저장하지 않아야 한다.
- 카탈로그 이미지 전체를 PWA 설치 시 프리캐시하지 않아야 한다.
- 카탈로그 메타데이터와 이미지 경로가 일치해야 한다.

### FR-032 사진 등록

- 사진 선택 즉시 원본 미리보기를 표시해야 한다.
- 원본은 자동 분석 결과와 별도로 보존해야 한다.
- 자동 분석은 배경 제거를 자동 확정하지 않고 초안을 생성해야 한다.
- 초안은 카테고리, 유형, 대표색, 주요색, 소재, 패턴과 보온 수준을 포함할 수 있다.
- 사용자가 수정한 값을 이후 자동 분석이 덮어쓰지 않아야 한다.

### FR-033 URL 등록

- 상품 페이지 URL과 직접 이미지 URL을 지원해야 한다.
- HTML 페이지는 `og:image`, JSON-LD Product, 본문 이미지 순서로 대표 이미지를 찾는다.
- 상품명과 이미지를 찾지 못하면 사진 업로드 또는 이미지 URL 입력으로 전환해야 한다.
- URL 이미지는 서버 프록시를 거쳐 브라우저 CORS 문제를 피해야 한다.
- 저장 항목에 출처 유형과 원본 URL을 기록해야 한다.

### FR-034 자동 분석 초안과 사용자 확정

- 모든 자동 분류와 색상 결과는 `draft` 상태여야 한다.
- 저장 전 사용자가 결과를 확인하고 수정할 수 있어야 한다.
- 사용자 확정값이 최종 권한을 가진다.
- 모델명, 버전, 처리 시각과 신뢰도 메모를 추적할 수 있어야 한다.
- 분석 실패가 의류 등록 실패를 의미해서는 안 된다.

### FR-040 일반 누끼

- 독립 ML 서비스는 `POST /api/background/remove` 계약을 제공해야 한다.
- 일반 상품컷의 배경을 제거하고 투명 배경 이미지를 반환해야 한다.
- 응답은 이미지, 크기, 바운딩 박스, 주요색, 모델과 버전을 포함해야 한다.
- 투명 영역을 제외한 의류 픽셀에서 주요색을 계산해야 한다.
- 실패 시 원본 이미지를 유지하고 다시 시도 또는 수동 등록을 제공해야 한다.

### FR-041 정밀 의류 추출

- 독립 ML 서비스는 `POST /api/clothing/extract` 계약을 제공해야 한다.
- 상의, 하의, 아우터, 신발, 가방, 액세서리 또는 자동 대상을 받을 수 있어야 한다.
- 응답은 추출 이미지와 분류·색상 초안을 포함해야 한다.
- 모델은 배포 단계 사전 준비를 우선하고 요청마다 다시 로드하지 않아야 한다.
- 모델 라이선스가 공개 서비스 사용을 허용하지 않으면 대체 모델을 선택해야 한다.

### FR-042 ML 상태와 대체 경로

- 프론트는 ML API 기준 URL을 환경 설정으로 받아야 한다.
- ML 서비스는 상태 확인 엔드포인트를 제공해야 한다.
- 연결 실패, 시간 초과, 모델 로드 중, 잘못된 이미지와 서버 오류를 구분해야 한다.
- 오류 후 원본 이미지로 수동 등록을 완료할 수 있어야 한다.
- 장애를 감추기 위해 성공처럼 보이는 가짜 분석 결과를 반환해서는 안 된다.

### FR-050 추천

- 추천은 퍼스널컬러 0.38, 날씨 0.22, 색상 조화 0.28, 착용 안정성 0.12의 기준 가중치를 사용해야 한다.
- 낮은 진단 신뢰도에서는 Top2 팔레트를 제한적으로 반영해야 한다.
- 기본 결과는 동일 의류가 겹치지 않는 최대 3개 조합이어야 한다.
- 각 결과에 네 축 점수와 자연어 근거를 표시해야 한다.
- 추천 불가 시 옷 부족, 카테고리 부족과 제외 상태 등의 원인을 표시해야 한다.
- 사용자는 목적과 대상 옷장을 선택할 수 있어야 한다.

### FR-060 저장 코디와 데일리룩

- 추천 조합을 제목, 의류 ID, 점수, 날씨, 모드, 근거와 저장 시각과 함께 저장해야 한다.
- 기본 폴더와 사용자 폴더를 지원해야 한다.
- 사용자 폴더는 생성, 이름 변경과 삭제가 가능해야 한다.
- 폴더 삭제 시 포함 코디는 기본 폴더로 이동해야 한다.
- 즐겨찾기, 검색과 정렬을 제공해야 한다.
- 데일리룩은 레이어 이동, 크기, 회전, 순서, 삭제, 배경과 텍스트를 지원해야 한다.
- 편집 상태를 저장·복원하고 1080×1440 PNG를 생성해야 한다.

### FR-070 브라우저 저장

- 진단 결과, 이력, 옷장 메타데이터, 저장 코디와 설정은 브라우저 저장소에 보존해야 한다.
- 큰 원본·누끼·완성 이미지는 IndexedDB에 저장해야 한다.
- localStorage에 대형 Base64 이미지를 누적하지 않아야 한다.
- 새로고침과 앱 재실행 후 데이터를 복원해야 한다.
- 전체 초기화는 명시적 확인 후 수행해야 한다.

### FR-080 PWA와 공개 접근

- 서비스는 HTTPS 공개 URL에서 동작해야 한다.
- 설치 가능한 PWA 매니페스트와 서비스 워커를 제공해야 한다.
- 앱 셸, 아이콘과 폰트는 프리캐시할 수 있다.
- 대형 카탈로그 전체는 프리캐시하지 않아야 한다.
- MediaPipe 자산은 첫 성공 이후 장기 캐시할 수 있다.
- 날씨와 외부 API는 제한된 캐시 정책을 사용해야 한다.

## 9. 시스템 책임 경계

### 9.1 Vercel 프론트와 경량 API

Vercel 배포는 다음 책임을 가진다.

- 카메라, 퍼컬 로컬 계산, 브라우저 저장과 추천 실행.
- 경량 URL 수집 API 제공 또는 프록시.
- ML API 기준 URL 환경 설정 제공.

### 9.2 URL 수집 API

URL 수집 API는 다음 책임을 가진다.

- 상품 페이지와 이미지 URL 수집.
- SSRF 방어와 리다이렉트 재검사.
- 대표 이미지와 상품명 추출.
- 이미지 바이트 프록시.
- 요청 크기, 응답 크기와 시간 제한.

### 9.3 독립 ML 서비스

ML 서비스는 다음 책임을 가진다.

- 일반 배경 제거와 정밀 의류 추출.
- 추출 영역 기반 대표색과 주요색 계산.
- 모델 상태와 버전 보고.
- 제한된 공개 Origin을 위한 CORS 정책.
- 모델 사전 준비와 프로세스 내 재사용.

### 9.4 금지되는 의존성

- 부모 저장소 파일을 상대 경로로 불러와서는 안 된다.
- 부모 폴더의 `node_modules`에 의존해서는 안 된다.
- 개발자 PC 절대 경로를 참조해서는 안 된다.
- 로컬 포트 `8001`을 운영 환경 고정 주소로 가정해서는 안 된다.
- 예외를 숨기고 핵심 라우트가 없는 상태로 정상 기동해서는 안 된다.

## 10. 데이터 계약 원칙

기존 `v2-schema-contract.md`를 기준으로 다음 원칙을 유지한다.

- 사람의 퍼스널컬러는 `PersonalSeasonId` 12계절 ID로 표현한다.
- 옷의 착용 적합성은 `WarmthLevel` 또는 WeatherBand 대응으로 분리한다.
- 대표색과 주요색은 HEX 저장값과 UI 색 이름을 분리한다.
- 자동 분석 결과는 `draft`, 사용자 확인 결과는 `confirmed`다.
- 사진, URL, 카탈로그 등록은 같은 최종 `ClothingItem` 계약으로 수렴한다.
- 원본, 누끼와 데일리룩 결과 이미지를 목적별로 분리한다.
- 기존 필드가 없어도 앱이 기동되는 하위 호환 마이그레이션을 사용한다.

## 12. 비기능 요구사항

### NFR-001 독립성

- 깨끗한 임시 폴더 또는 CI에서 `npm ci`, 타입 검사, 테스트와 빌드가 성공해야 한다.
- Git 추적 파일만 체크아웃한 상태에서 동일한 검증이 성공해야 한다.
- ML 서비스도 자체 의존성 파일과 진입점을 가져야 한다.

### NFR-002 성능과 가용성

- 앱 셸은 의미 있는 로딩 상태를 즉시 표시해야 한다.
- 대형 카탈로그는 초기 번들에 포함하지 않아야 한다.
- 메인 JavaScript 번들은 기능 단위 코드 분할을 검토해야 한다.
- 얼굴 분석은 프레임마다 React 전체 상태를 갱신하지 않아야 한다.
- ML 요청 전 이미지 크기를 제한해야 한다.
- ML 장애가 진단, 날씨, 옷장 열람과 수동 등록을 막지 않아야 한다.
- 날씨 장애가 수동 기온 추천을 막지 않아야 한다.

### NFR-003 보안

- URL 수집은 HTTP와 HTTPS만 허용해야 한다.
- localhost, 사설 IP, link-local, 예약 주소와 리다이렉트 우회를 차단해야 한다.
- 응답 크기, 리다이렉트 횟수와 시간을 제한해야 한다.
- 업로드 파일의 MIME과 실제 디코딩 가능 여부를 검사해야 한다.
- CORS는 필요한 프론트 Origin만 허용해야 한다.
- 서버 로그에 얼굴 이미지나 전체 사용자 입력을 남기지 않아야 한다.
- 비밀 값은 환경변수로 관리해야 한다.

### NFR-004 개인정보

- 얼굴 분석은 기본적으로 브라우저 안에서 처리해야 한다.
- 얼굴 사진을 의류 ML 서버로 전송해서는 안 된다.
- 의류 사진 전송 전 사용자 행동이 있어야 한다.
- 서버는 의류 이미지를 영구 저장하지 않는 것을 기본으로 한다.
- 사용자 데이터 초기화 기능을 제공해야 한다.

### NFR-005 접근성과 호환성

- 키보드만으로 핵심 흐름을 진행할 수 있어야 한다.
- 아이콘 단독 버튼은 접근 가능한 이름을 가져야 한다.
- 모달과 시트는 포커스 관리, Escape와 닫기 행동을 제공해야 한다.
- 색상 정보는 색만으로 의미를 전달하지 않아야 한다.
- 390px 모바일에서 의도하지 않은 문서 수평 스크롤이 없어야 한다.
- 최신 Chrome, Edge, Android Chrome을 우선 지원한다.
- iOS Safari의 카메라, PWA와 다운로드를 별도 검증한다.

### NFR-006 관측 가능성

- 프론트는 기능별 실패 유형을 구분 가능한 오류 코드로 기록할 수 있어야 한다.
- ML 서비스는 상태, 모델 로드 여부, 처리 시간과 오류 유형을 제공해야 한다.
- 로그에 이미지 본문과 민감한 URL 쿼리 값을 남기지 않아야 한다.

## 13. 성공 지표

1. 독립 저장소 CI 성공률이 100%여야 한다.
2. 신규 사용자 첫 진입의 가짜 진단 결과 노출이 0건이어야 한다.
3. 지원 이미지 기준 일반 누끼 성공률이 검증 세트의 90% 이상이어야 한다.
4. ML 장애 상황에서 수동 의류 등록 완료율이 100%여야 한다.
5. 프리셋 옷장 기준 겹치지 않는 추천 3개 생성 성공률이 100%여야 한다.
6. 새로고침 후 진단, 옷장과 저장 코디 복원 성공률이 100%여야 한다.
7. P0 브라우저 콘솔 오류가 0건이어야 한다.

누끼 품질 지표는 모델과 검증 이미지 세트가 확정된 뒤 유형별로 세분화한다.

## 14. 출시 인수 기준

### AC-01 독립 프론트

- Git 추적 파일만 새 폴더에 체크아웃한다.
- `npm ci`, 타입 검사, 전체 테스트와 프로덕션 빌드가 성공한다.
- 부모 폴더에 다른 `node_modules`가 없어도 성공한다.

### AC-02 실제 진단

- 저장소가 비어 있는 새 브라우저에서 미진단 상태로 시작한다.
- 카메라 또는 업로드 사진과 설문을 완료한다.
- 계산된 Top1, Top2, 신뢰도와 근거가 저장된다.
- 새로고침 후 같은 결과가 복원된다.

### AC-03 날씨

- 위치 허용 시 현재 위치 기반 날씨가 표시된다.
- 위치 거부 시 서울 기준임이 표시된다.
- 수동 기온 변경 후 추천에 해당 값이 반영된다.

### AC-04 의류 등록

- 원본 사진을 선택하면 즉시 미리보기가 보인다.
- ML 정상 상태에서 분석 초안과 누끼 결과가 반환된다.
- 사용자가 값을 수정해 저장할 수 있다.
- ML 중단 상태에서도 원본과 수동 정보로 저장할 수 있다.
- 공개 URL에서 대표 이미지와 상품명 초안을 가져올 수 있다.
- 사설 주소와 리다이렉트 우회가 차단된다.

### AC-05 추천과 저장

- 충분한 옷장에서 겹치지 않는 추천 3개를 생성한다.
- 네 축 점수와 근거가 표시된다.
- 추천을 폴더에 저장하고 즐겨찾기로 분류할 수 있다.
- 데일리룩 상태가 복원되고 1080×1440 PNG가 생성된다.

### AC-06 공개 배포

- Vercel 공개 URL을 PC와 모바일 외부 기기에서 열 수 있다.
- HTTPS 카메라 권한이 정상 동작한다.
- 프론트가 운영 ML URL을 사용한다.
- 운영 누끼 API가 고정 503이 아니라 실제 처리 또는 명확한 장애 상태를 반환한다.

| 데모 데이터 재유입 | 사용자 신뢰 훼손 | 신규 상태 회귀 테스트 |

## 17. 미결정 사항

1. ML 서비스 1차 호스팅 제공자를 결정해야 한다.
2. 정밀 추출 모델의 공개 서비스 라이선스와 품질을 검증해야 한다.
3. ML API를 Vercel이 프록시할지 브라우저가 직접 호출할지 결정해야 한다.
4. 130MB 이상 카탈로그를 정적 자산에 유지할지 외부 저장소로 옮길지 결정해야 한다.
5. 설문 전용 진단을 1.0에서 허용할지 사진을 필수로 할지 결정해야 한다.
6. 운영 오류 관측 도구 도입 범위를 결정해야 한다.
