# FRD — 퍼스널컬러 옷장·코디 추천 서비스, 제로베이스 구축 명세

작성 기준일 2026-07-06. 이 문서는 **현 저장소와 그 산출물이 전혀 없다고 가정하고**, 같은 제품을 처음부터 만들기 위해 필요한 요구사항을 리버스 엔지니어링으로 추출한 명세다. 추출 원본은 2026-07-06 기준 v2 구현(테스트 129개 통과, 프로덕션 빌드 통과)과 v1에서 검증된 로직 자산이다.

**관계 문서.** `리버스frd.md`(2026-07-02)는 "v1을 해부하고 v2로 이행하는" 문서이고, 본 문서는 "아무것도 없는 팀이 이 제품을 만들 수 있는가"를 기준으로 쓴 무전제 명세다. 이행 이력·승계/폐기 판단은 그쪽을, 제품 자체의 계약은 이쪽을 본다.

**읽는 법.** §3의 각 기능 요구(FR)는 요구 서술과 수용 기준으로 구성된다. 수치·가중치·키 이름 등 구현에서 실증된 초기값은 부록 A에 모았다. 본문의 수치는 "검증된 초기값"이며 절대 상수가 아니다 — 바꾸려면 근거와 회귀 테스트를 함께 바꾼다.

## 1. 제품 정의

### 1.1 한 줄 정의

얼굴 사진과 설문으로 12계절 퍼스널컬러를 진단하고, 사용자의 디지털 옷장·오늘의 날씨·색채 조화를 함께 반영하여 **근거를 설명할 수 있는** 코디를 추천하고, 그 결과를 데일리룩 이미지로 저장·내보내는 웹 서비스.

### 1.2 가치 가설

1. **통합이 차별점이다.** 퍼스널컬러 진단, 디지털 옷장, 코디 추천은 각각 존재하지만 "내 톤 → 내 옷 → 오늘 날씨 → 오늘 입을 옷"이라는 하나의 의사결정 흐름으로 묶은 서비스는 드물다.
2. **진입 장벽을 데이터 확보 경로 3개로 흡수한다.** 검수된 프리셋 카탈로그, 쇼핑몰 URL 수집, 직접 촬영. "옷을 하나하나 찍어야 한다"는 디지털 옷장의 고질적 장벽을 앞의 두 경로가 흡수한다.
3. **추천은 블랙박스가 아니다.** 모든 추천 점수는 4축(퍼스널컬러 적합·날씨·색 조화·안정성)으로 분해되어 사용자에게 근거 문장으로 제시된다.
4. **로컬 우선.** 얼굴 사진 분석과 추천 계산은 전부 브라우저 안에서 일어난다. 얼굴 이미지는 서버로 전송하지 않는다. 이것은 프라이버시 강점이자 운영비 0원의 구조적 근거다.

### 1.3 대상 사용자와 이용 환경

- 자신의 퍼스널컬러를 알고 싶고, 보유 옷 활용도를 높이고 싶은 일반 사용자. 전문 지식 전제 없음.
- 모바일·데스크톱 브라우저(최신 Chromium/Safari 계열). 카메라·위치 권한은 선택이며 거부해도 서비스가 성립해야 한다.
- 계정 없음. 모든 데이터는 사용자의 기기(브라우저 저장소)에 남는다.

### 1.4 성공 기준 — 골든 패스

아래 단일 시나리오가 외부 ML 서버 없이, 새 브라우저 프로필에서 5분 안에 끊김 없이 완주되면 MVP 성공이다.

1. 퍼스널컬러 진단을 완료하고 12계절 결과(Top1/Top2, 신뢰도, 팔레트, 근거)를 확인한다.
2. 프리셋 카탈로그에서 옷장을 선택(또는 생성)한다.
3. 서로 겹치지 않는 코디 추천 3개를 받고, 각 추천의 근거 문장을 읽는다.
4. 추천 하나를 데일리룩으로 저장하고 캔버스에서 배치를 편집한다.
5. 데일리룩을 PNG 파일로 내보낸다.

### 1.5 비목표

- 생성형 가상 착용(VTON), 사이트별 전용 스크래퍼, 소셜 기능, 쇼핑 연동·구매 유도.
- 서버 측 사용자 DB와 계정 시스템(초기 버전에서는 명시적으로 제외 — §10 열린 결정).
- 추천 가중치의 온라인 학습(분석·실험으로만 수행하고 서비스 기능화하지 않는다).

## 2. 화면과 사용자 여정

### 2.1 화면 맵

UI는 단일 페이지 앱이며 화면 전환은 앱 내부 상태로 관리한다. 화면 구성은 다음과 같다.

| 화면 | 하위 뷰 | 사용자가 하는 일 |
| --- | --- | --- |
| 홈 `home` | — | 대시보드. 옷장 현황, 오늘 날씨, 최근 데일리룩, 각 기능 빠른 진입 |
| 퍼스널컬러 `personal` | `photo` → `questionnaire` → `result` | 사진 분석 → 설문 응답 → 진단 결과 확인·저장 |
| 옷장 `wardrobe` | `list` / `detail` / `catalog` / `manual` | 옷장 생성·삭제, 의류 목록·상세, 카탈로그 다중 선택, 수동 등록(사진/URL) |
| 추천 `recommend` | — | 날씨·모드 기반 코디 추천 확인, 저장, 앵커(기준 옷) 추천 |
| 저장 코디 `saved` | — | 저장한 코디 목록, 삭제, 데일리룩 편집 진입 |
| 데일리룩 편집 `tryon` | — | 캔버스에서 의류 레이어 배치·편집, 확정, PNG 내보내기 |
| 설정 `settings` | — | 퍼스널컬러 재측정 진입, 전체 데이터 초기화 |

### 2.2 핵심 여정

**여정 A — 진단.** 홈 → 사진 촬영/업로드(건너뛰기 가능) → 설문 → 결과 저장 → 홈 복귀. 사진 없이 설문만으로도 결과가 나와야 한다.

**여정 B — 옷장 구축.** 옷장 목록 → 카탈로그에서 다중 선택 → 기존 옷장에 추가 또는 새 옷장 생성. 부가 경로로 사진 업로드 등록(FR-6), URL 수집 등록(FR-7).

**여정 C — 추천과 데일리룩.** 추천 화면 → 겹치지 않는 조합 3개 확인 → 저장 → 데일리룩 편집 → PNG 내보내기.

### 2.3 내비게이션 규칙

- URL 경로는 사용하지 않는다(항상 `/`). 화면 상태는 메모리에 있고 브라우저 히스토리(pushState/popstate)와 연동해 뒤로가기를 지원한다.
- 새로고침 시 화면은 홈으로 복귀하되, 저장된 데이터(진단 결과·옷장·코디)는 전부 복원되어야 한다.
- 별도 서버 라우팅·SPA rewrite가 필요 없는 구조를 유지한다.

## 3. 기능 요구사항

### FR-1 퍼스널컬러 진단 — 사진 분석

- 카메라 촬영 또는 이미지 업로드로 얼굴 사진 1장을 입력받는다.
- 얼굴 랜드마크 검출은 온디바이스로 수행한다(검증 구현은 MediaPipe Face Landmarker, wasm+모델 CDN 로드, GPU 실패 시 CPU 폴백). 얼굴 이미지·파생 픽셀 데이터를 서버로 전송해서는 안 된다.
- 랜드마크에서 피부·머리카락·눈·입술 ROI를 샘플링하고, 흰 종이/배경 등 기준 흰색을 이용한 다단계 조명 보정을 거쳐 RGB/HSL/Lab 특징을 계산한다.
- 사진 품질(조명, 얼굴 크기, 검출 신뢰도)을 점수화한다. 이 품질 점수는 FR-3의 융합 가중에 쓰인다.
- 얼굴 검출 실패, 카메라 권한 거부, CDN 로드 실패 시에도 진단 흐름이 끊기면 안 된다 — 설문 단독 경로로 자동 강등한다.

수용 기준. 정면·자연광 사진에서 피부/머리/눈/입술 대표색이 추출되고, 사진을 건너뛰어도 결과 화면까지 도달한다.

### FR-2 퍼스널컬러 진단 — 설문

- 설문은 4축(온도 temperature, 명도 lightness, 청탁 clarity, 대비 contrast)을 모두 커버하는 문항 세트로 구성한다. 문항·색 스와치·축별 가중치는 코드가 아니라 데이터 계약(상수 테이블)으로 관리한다.
- 소요 시간 1분 내를 기준으로 문항 수를 정한다(검증 구현 이력에서 5~12문항 사이를 오갔다 — 문항 수 자체는 계약이 아니고 4축 커버리지가 계약이다).
- 각 응답은 4축 점수 벡터로 환산되어 누적된다.

수용 기준. 설문만으로 12계절 중 하나가 Top1으로 결정되고, 축별 기여가 결과 화면 근거에 나타난다.

### FR-3 진단 융합과 결과

- 사진 시즌 점수는 팔레트 거리 42% + 특징 유사도 58%로 합성한다(검증 초기값).
- 사진과 설문의 융합 가중은 사진 품질에 따라 동적으로 정하되 사진 비중을 22%~36%로 clamp한다. 설문이 안정화 역할을 한다는 설계 의도를 유지한다.
- 결과는 다음을 반드시 포함한다 — Top1/Top2 시즌(12계절 ID와 한글명), 신뢰도(0~1), 판정 근거 타입(hybrid/photo/questionnaire), 근거 상세(사진 신호, 설문 신호, 일관성, 융합 가중, 경계 여부), 추천 팔레트(HEX 배열), 추출 색(피부/머리/눈/입술), 설명 문장.
- Top1과 Top2 점수 차가 작으면 "경계 판정"으로 표시하고 두 시즌을 함께 안내한다.
- 결과와 진단 이력은 기기 저장소에 영속화하고, 이력에서 과거 결과를 다시 적용할 수 있다.

수용 기준. 같은 입력에 같은 결과(결정적 계산). 결과 화면에서 신뢰도·근거·팔레트가 보이고, 새로고침 후에도 유지된다.

### FR-4 디지털 옷장 관리

- 옷장은 복수 생성·삭제할 수 있고, 각 의류는 정확히 하나의 옷장에 속한다.
- 의류는 단일 색상이 아니라 구조화 속성으로 저장한다 — 대표색(이름+HEX), 상위 3색과 비율, 카테고리(상의/하의/아우터/신발/액세서리), 세부 타입, 재질, 패턴, 데님 여부·워시, 착용 계절 태그, 가용 상태(보유중/세탁중/보관중/추천제외), 중립색 여부, 출처(catalog/upload/url). 전체 스키마는 §5.1.
- 옷장 상세에서 각 의류의 현재 퍼스널컬러 기준 적합 점수를 보여준다.
- 의류 이미지 원본과 컷아웃은 기기 저장소에 저장한다(저장 전략은 §5.5).

수용 기준. 의류 100벌 규모에서 목록·상세·삭제가 지연 없이 동작하고, 새로고침 후 이미지까지 복원된다.

### FR-5 프리셋 카탈로그

- 검수된 의류 카탈로그를 앱에 내장한다(검증 구현 규모는 890벌 — 상의·하의 등 860 + 아우터 30, 이미지 861장).
- 카탈로그 큐레이션 계약. **어떤 퍼스널컬러 × 어떤 기온 밴드에서도 겹치지 않는 추천 3개가 성립**하도록 색·카테고리·warmth 분포를 설계한다. 이것은 UI가 아니라 데이터 설계 요구다.
- 카탈로그 메타데이터는 실측값이어야 한다 — 대표색은 이미지 픽셀에서 추출·검수하고, 중립색 여부·재질·패턴을 사람이 확인한다. 생성 스크립트는 저장소 안에서 재실행 가능해야 한다(원본 자산 경로가 특정 컴퓨터에 묶이면 안 된다).
- 이미지는 데일리룩 캔버스에 올릴 수 있는 품질(투명 배경 누끼, 충분한 해상도)을 기준으로 한다.

수용 기준. 12계절 × 전 기온 밴드 조합에서 추천 3개 성립을 자동 테스트로 증명한다.

### FR-6 의류 등록 — 사진 업로드

- 사진을 업로드하면 **원본 사진을 그대로 등록**한다. 누끼(배경 제거)는 자동 적용하지 않고 사용자가 버튼으로 선택했을 때만 적용한다.
- 색상·분류 자동 조사는 업로드 즉시 기본 실행한다 — 대표색·상위 색, 카테고리·타입 초안, 착용 계절·재질 예측 초안, "AI 분석" 배지.
- **초안-확정 원칙(§4.5).** 자동 인식 결과는 전부 초안이며 사용자가 폼에서 수정한 값이 최종 권한이다. 저장된 아이템은 항상 확정값이다.
- 자동 분석 실패(서버 부재 포함) 시에도 수동 입력만으로 등록이 완료되어야 한다.

수용 기준. ML 서버가 꺼진 상태에서 사진 등록이 성립한다. 서버가 켜진 상태에서는 등록 직후 미리보기가 원본이고, 누끼 버튼을 눌렀을 때만 컷아웃으로 교체된다.

### FR-7 의류 등록 — URL 수집

- 사용자가 상품 페이지 또는 이미지 URL을 붙여넣으면 대표 이미지와 상품명을 찾아 등록 초안을 만든다.
- 브라우저는 외부 이미지를 CORS 때문에 직접 분석할 수 없으므로, 수집은 반드시 서버 프록시를 거친다. 프록시는 두 엔드포인트다 — URL 분석(HTML이면 대표 이미지·상품명 파싱, 이미지면 그대로), 이미지 바이트 프록시.
- 대표 이미지 탐색 우선순위는 `og:image` → JSON-LD Product → 본문 이미지 휴리스틱 순이다.
- 상품명은 사이트 꼬리표(`| 무신사`, `- 사이즈 & 후기` 류)를 제거해 브랜드 필드의 초안으로 넣는다.
- 수집된 초안은 사진 업로드와 동일한 자동 분석·등록 경로를 공유한다(별도 저장 경로를 만들지 않는다). 저장 아이템에 출처 `sourceType: 'url'`과 원본 주소 `sourceRef`를 기록한다.
- 프록시는 SSRF 가드를 통과해야만 요청을 수행한다(규칙은 §6.3).

수용 기준. 주요 쇼핑몰 상품 URL에서 대표 이미지·상품명 초안 생성 → 수정 → 옷장 저장까지 완주. 사설 IP·리다이렉트 우회 시도는 서버 테스트로 차단을 증명한다.

### FR-8 누끼·색 추출 파이프라인 (선택 의존성)

골든 패스 밖의 부가 능력이다. 이 파이프라인이 전부 꺼져도 FR-1~7, 9~12는 성립해야 한다.

- **일반 누끼.** 상품컷 대상 경량 배경 제거(검증 구현은 rembg u2netp, 첫 요청 시 lazy load). 응답은 투명 PNG data URL + 의류 bbox + 대표색 최대 5개.
- **정밀 추출.** 착용샷에서 부위(상의/하의/아우터/신발/가방/auto)를 분리(검증 구현은 SegFormer b3-fashion). 카테고리 자동 감지, 세부 라벨, 착용 계절·재질 예측을 동반한다. **주의 — 검증 구현의 모델은 비상업 연구용 라이선스를 승계하므로 공개 상용 서비스에는 쓸 수 없다. 실서비스에는 상용 가능 모델로 대체하거나 이 경로를 제외한다.**
- **대표색 추출.** 알파 마스크로 배경 픽셀을 제외하고 추출한다. 즉 색상 자동 조사도 내부적으로 세그멘테이션을 쓰지만, 그 컷아웃을 화면에 자동 반영할지는 별개 결정이다(FR-6의 원본 유지 원칙).
- 착용 계절 예측의 참고 성능(검증 구현) — 라벨 516장, 95특징 Random Forest, 정확도 89.74%(룰 베이스라인 19.2%). 단 상품컷에는 착용샷 전제 특징이 무의미해지는 계통 편향이 확인되어 있으므로, 재구축 시 입력 유형(착용샷/상품컷)을 구분해 설계한다.

수용 기준. 서버 헬스체크(2.5초 타임아웃)로 가용성을 감지해 UI가 기능을 감추거나 안내한다. 서버 미가동 시 등록 흐름 무중단.

### FR-9 코디 추천 엔진

- 추천 점수는 4축 가중합이다 — 퍼스널컬러 적합 0.38, 날씨 적합 0.22, 색 조화 0.28, 안정성 0.12(검증 초기값).
- **신뢰도 적응 가중.** 진단 신뢰도가 낮으면 퍼스널컬러 축 비중을 내리고 조화 축을 올린다.
- 퍼스널컬러 적합은 Top1 팔레트를 기본으로 하되 Top2 팔레트를 최대 35%까지 혼합한다. 색 거리 계산은 §4.2의 색 수학 계약을 따른다.
- 색 조화는 Itten 색상환 기반 배색 분류(단색/유사/긴장/삼색/보색, 기본점 88/82/80/76/55)와 명도·채도 균형의 가중합(색상 0.5, 명도 0.3, 채도 0.2)으로 계산한다. 중립색은 배색 판정에서 별도 취급한다.
- 결과는 **서로 겹치지 않는 조합 3개**(카테고리 구성이 유효한 상·하의+α)이며, 각 조합에 점수와 근거 문장 목록을 붙인다. 추천 모드(데일리/출근/데이트/발표)와 기온 밴드를 입력 조건으로 받는다.
- **앵커 추천.** 기준 옷 1벌을 고정하고 나머지를 추천한다. 탐색 공간을 가지치기하고(검증 구현에서 43,650→3,600 조합) 같은 상대 옷의 등장을 2회로 제한해 다양성을 확보한다.
- 모든 매직넘버는 이름 있는 상수 + 근거 주석으로 관리하고, 점수 회귀 스냅샷 테스트로 보호한다.

수용 기준. 데모 옷장에서 추천 3개가 항상 성립하고 아이템이 중복되지 않는다. 가중치 변경 시 회귀 테스트가 깨져서 변경 사실을 드러낸다.

### FR-10 날씨 연동

- 위치 권한을 받아 현재 기온·체감·강수·대기질을 조회하고, 위치명을 한국어로 역지오코딩한다. 검증 구현의 공급자는 Open-Meteo(예보·대기질, 무키)와 BigDataCloud 역지오코딩(무키)이다.
- 권한 거부·API 실패 시 서울 기준(37.5665, 126.978, 라벨 "서울 기준")으로 자동 폴백하고 그 사실을 표시한다.
- 기온은 추천 엔진이 쓰는 이산 기온 밴드(WeatherBand)로 매핑한다. 밴드 경계는 데이터 계약으로 관리한다.

수용 기준. 오프라인·권한 거부 상태에서도 추천 화면이 서울 폴백으로 성립한다.

### FR-11 데일리룩 캔버스와 PNG 내보내기

- 저장 코디를 1080×1440 캔버스에 의류 레이어로 자동 배치한다(카테고리별 기본 위치·크기).
- 레이어는 드래그 이동·스케일·회전·순서 변경이 가능하고 텍스트 레이어를 추가할 수 있다.
- "확정" 시 캔버스를 렌더링한 최종 이미지를 코디에 저장하고, PNG 파일로 다운로드할 수 있다.
- 캔버스 상태(레이어 변환값 전체)는 코디에 영속화되어 재편집이 가능하다.

수용 기준. 저장 → 편집 → 확정 → PNG 다운로드 → 재진입 시 이전 배치 복원.

### FR-12 저장 코디, 홈, 설정

- 저장 코디 목록에서 점수·기온 밴드·모드·저장 시각을 보여주고, 삭제와 데일리룩 편집 진입을 제공한다.
- 홈은 진단 결과 요약, 옷장 통계, 오늘 날씨, 최근 데일리룩, 기능 진입 카드를 보여준다.
- 설정에서 퍼스널컬러 재측정 진입과 전체 데이터 초기화(로컬 저장소 완전 삭제)를 제공한다. 초기화는 파괴적 동작이므로 확인 단계를 거친다.

## 4. 도메인 규칙 (계약)

### 4.1 12계절 체계와 명칭 분리 원칙

- 퍼스널컬러 시즌은 12계절 ID로 고정한다 — `light-spring` `true-spring` `bright-spring` / `light-summer` `true-summer` `soft-summer` / `soft-autumn` `true-autumn` `dark-autumn` / `dark-winter` `true-winter` `bright-winter`.
- **사람의 퍼스널컬러 "시즌"과 옷의 "착용 계절"(봄/가을·여름·겨울·사계절)은 타입과 문구에서 분리한다.** 같은 "계절"이라는 단어가 두 개념을 오가며 생기는 혼용이 v1의 실증된 결함이었다. 코드 타입명·UI 문구 모두에서 구분한다.
- 12계절 각각의 팔레트(추천 HEX 목록), 회피색, traits(온도/명도/청탁/대비 프로필), 한글 별칭·설명은 단일 워크북 데이터로 관리한다.

### 4.2 색 수학

- 변환 체인은 HEX ↔ sRGB ↔ Lab(D65) ↔ LCh를 표준으로 한다.
- 지각적 색 거리는 CIEDE2000(Sharma 2005 구현 기준)을 쓴다. 단순 RGB 유클리드 거리를 지각 판단에 쓰지 않는다.
- 명도·채도 판단은 Lab의 L, LCh의 C를 기준으로 한다. HSL은 UI 보조용으로만 쓴다.

### 4.3 시즌 스펙트럼과 적합도

- 각 시즌 팔레트를 LCh 공간의 스와치 집합으로 표현하고, 임의 색의 시즌 적합도를 스와치 거리 기반 멤버십 함수로 계산한다.
- 회피색에 근접한 색은 감점한다.
- 의류의 적합도는 대표색 하나가 아니라 상위 색 비율을 가중해 계산한다.

### 4.4 의류 속성 파생

- **warmth(보온 수준).** 카테고리·재질·타입에서 파생하는 이산 레벨이며, 기온 밴드와의 매칭에 쓴다. 파생 규칙은 순수 함수로 두고 테스트한다.
- **중립색.** 흰/검/회/베이지 계열은 `isNeutral: true`로 표시하고 배색 판정에서 페널티 없이 조합을 허용한다.
- **데님.** 이름 신호("청", "denim")를 우선하고, 없으면 HEX 밝기 기반 규칙으로 워시(light/mid/dark/black)를 추정한다.
- 이름 신호(자유 텍스트)를 읽는 분류 입력은 명시적으로 한 곳(브랜드/이름 필드)으로 제한한다.

### 4.5 초안-확정 원칙

자동 인식(색·카테고리·계절·재질·상품명 파싱)의 모든 출력은 **초안**이다. 사용자가 폼에서 확인·수정한 값이 **최종 권한**이며, 저장된 아이템은 항상 확정값이다. 이후 어떤 자동 처리도 사용자가 확정한 필드를 덮어쓰면 안 된다(v1의 실증 버그 — 사용자 계절 수정 무시 — 의 재발 방지 계약).

- 모든 읽기는 try-catch + 기본값 폴백으로 감싼다(손상 데이터가 앱을 죽이면 안 된다).

## 6. 시스템 아키텍처와 API 계약

### 6.1 구성

시스템은 세 덩어리이며 결합도는 아래로 갈수록 낮다.

1. **클라이언트 SPA** (React 19 + TypeScript + Vite + Tailwind 4). 진단·추천·옷장·캔버스 전부 온디바이스 계산. 상태는 도메인별 훅(진단/옷장/저장코디/날씨/라우팅/수동등록)으로 분리.
2. **경량 수집 API** (무상태, 검증 구현은 FastAPI + httpx). URL 수집 프록시 2개 + 헬스체크. ML 의존성 없음 — 서버리스로 이식 가능한 크기를 유지한다.
3. **ML API** (선택, 무상태). 누끼·정밀 추출·계절/재질 예측. 무겁고(torch 계열) 콜드스타트가 길므로 별도 배포 단위로 격리한다. 부재 시 앱은 기능 강등으로 대응한다.

### 6.2 API 계약

모든 응답은 JSON(이미지 프록시 제외), 실패는 4xx/5xx + 메시지. 클라이언트 호출은 same-origin 상대경로 `/api/...`를 기본으로 한다.

| 엔드포인트 | 요청 | 응답 | 소속 |
| --- | --- | --- | --- |
| `GET /api/health` | — | { ok, model, modelLoaded } — 클라이언트 타임아웃 2.5초 | 수집/ML |
| `POST /api/ingest/url` | { url } | 이미지 URL이면 그 자체, HTML이면 대표 이미지 후보 + 상품명 | 수집 |
| `POST /api/ingest/image` | { url } | 이미지 바이트(PNG/JPEG) — 브라우저 CORS 우회용 프록시 | 수집 |
| `POST /api/background/remove` | multipart { file } | { imageDataUrl(투명 PNG), bbox, colors(≤5), model, version, width, height, processedAt } | ML |
| `POST /api/clothing/extract` | multipart { file, targetPart } | 위 응답 + category, fineLabels, predictedSeasonTag, predictedMaterial | ML |

### 6.3 SSRF 가드 (수집 API 필수 계약)

- http/https 스킴만 허용한다.
- 호스트를 IP로 해석해 사설 대역·localhost·link-local·메타데이터 주소를 차단한다.
- 리다이렉트는 최대 4회 추적하되 **매 홉마다 재검사**한다.
- 응답 크기 상한 8MB. 타임아웃 설정 필수.
- 검사 로직은 공용 함수 한 곳으로 유지한다(엔드포인트마다 복제 금지).

### 6.4 외부 서비스 의존성

| 의존성 | 용도 | 키 | 실패 시 |
| --- | --- | --- | --- |
| MediaPipe wasm+모델 CDN (jsdelivr, storage.googleapis.com) | 얼굴 랜드마크 | 불필요 | 설문 단독 진단으로 강등 |
| api.open-meteo.com / air-quality-api.open-meteo.com | 날씨·대기질 | 불필요 | 서울 폴백 |
| api.bigdatacloud.net reverse-geocode-client | 위치명 한글화 | 불필요 | 좌표 라벨 생략 |

전부 무키·무료라 환경변수 없이 동작하는 것이 계약이다. 유료 키가 필요한 의존성을 추가할 때는 NFR-1(로컬 우선)과 충돌 여부를 검토한다.

## 7. 비기능 요구사항

**NFR-1 프라이버시.** 얼굴 사진과 그 파생 데이터는 기기를 떠나지 않는다. 의류 이미지는 사용자가 명시적으로 트리거한 기능(누끼, URL 수집)에서만 서버를 왕복하고 서버에 저장하지 않는다(무상태).

**NFR-2 성능.** ML 모델(브라우저 얼굴 모델, 서버 누끼/세그멘테이션)은 전부 첫 사용 시 lazy load. 첫 진단 준비 3~5초, 누끼 첫 요청의 모델 로드 수십 초는 진행 상태 UI로 흡수한다. 메인 번들 초기 로드는 gzip 300KB 이하를 예산으로 한다.

**NFR-3 가용성·폴백.** 외부 의존성(§6.4)과 선택 서버(ML)는 전부 "없어도 성립"이 기본자세다. 골든 패스는 어떤 외부 서버도 없이 완주 가능해야 한다.

**NFR-4 저장 한계.** 이미지 오프로드(§5.5)로 localStorage 5MB 제약을 우회하되, IndexedDB 용량 초과 시 사용자에게 정리 안내를 제공한다. 저장 실패가 조용히 데이터를 유실하면 안 된다.

**NFR-5 라이선스.** 배포 형상에 포함되는 모든 모델·라이브러리는 상용 배포 가능 라이선스여야 한다. 비상업 연구용 모델(검증 구현의 정밀 추출 모델이 해당)은 공개 서비스 빌드에서 제외한다.

**NFR-6 테스트.** 추천 점수·조화 점수는 회귀 스냅샷으로 고정한다. 저장 계층은 fake-indexeddb로, 수집 API는 SSRF 차단 케이스 포함 서버 테스트로 검증한다. 타입체크(tsc --noEmit)와 전체 테스트가 CI 게이트다(검증 구현 규모 — 프론트+서버 테스트 129개+17개).

**NFR-7 한국어·인코딩.** UI 문자열은 UTF-8 한국어. 소스·문서 전반에서 인코딩 깨짐을 CI 수준에서 감시한다(v1의 실증 문제). 개발 환경이 한글 경로에 있어도 도구가 동작해야 한다.

**NFR-8 보안.** 수집 프록시의 SSRF 가드(§6.3)는 기능이 아니라 보안 계약이다. CORS는 개발에서만 전체 허용하고 프로덕션은 서비스 도메인으로 제한한다.

## 8. 배포 요구사항 (Vercel 기준 실서비스)

### 8.1 배포 형상

| 구성요소 | 배포 위치 | 근거 |
| --- | --- | --- |
| SPA + 카탈로그 정적 자산 | Vercel 정적 호스팅 | 빌드 산출물 + 이미지 861장(약 133MB), rewrite 불요(URL 경로 미사용), HTTPS 자동 — 카메라·위치 권한 성립 |
| 수집 API (ingest 2 + health) | Vercel Functions(Python 또는 TS 재작성) | 의존성 경량(httpx 수준), 무상태 — 서버리스 적합 |
| ML API | Vercel 밖(별도 호스트) 또는 제외 | torch·transformers 수 GB — 서버리스 번들 한도(기본 250MB, Python 상향 옵션 있음) 초과, 콜드스타트 부적합 |

### 8.2 필수 조치

1. 저장소 위생 — node_modules·빌드 산출물을 git 추적에서 제외하고, 배포 대상 앱 디렉터리를 프로젝트 Root로 지정한다. 카탈로그 이미지는 반드시 git에 포함한다(빌드 입력).
2. API 베이스 전략 — 클라이언트는 same-origin `/api` 유지, 외부 ML 호스트를 쓸 경우 rewrite 또는 `VITE_API_BASE` 환경변수로 주입한다.
3. CORS를 서비스 도메인으로 제한하고, 로컬호스트 전제 안내 문구(카메라 권한 에러 메시지)를 실제 도메인 기준으로 수정한다.
4. NFR-5 라이선스 게이트 통과(정밀 추출 모델 제외 또는 교체).

### 8.3 ML 경로 3택 (실서비스 결정 사항)

- A. 별도 호스트(HF Spaces/Cloud Run/Railway 등)에 ML API 배포 + 도메인 제한 CORS. 기능 전부 유지, 콜드스타트·비용 관리 필요.
- B. 일반 누끼를 클라이언트 WASM(브라우저 내 배경 제거)으로 대체. 서버 완전 제거, 정밀 부위 분리는 포기.
- C. 초기 출시에서 누끼 제외. 원본 사진 + 색 자동 조사(A/B 중 택일 시)만으로 등록 성립 — 등록의 "원본 유지, 누끼는 선택" 설계가 이 강등을 이미 허용한다.

### 8.4 용량·비용 가드레일

- 정적 자산 133MB/861장은 배포 한도 내이지만 무료 티어 대역폭(월 100GB 수준)을 고려해 카탈로그 이미지의 WebP 변환·지연 로드를 권장한다.
- 수집 프록시는 요청당 8MB 상한이 있어 함수 실행 시간·메모리 기본값으로 충분하다.
