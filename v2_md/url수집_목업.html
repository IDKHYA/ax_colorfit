붙여넣은 주소의 이미지는 내 브라우저에만 저장되고 어디에도 재배포되지 않습니다.

구현 메모 — 화면 1

  • 기존 "나만의 옷 추가" 화면에 세 번째 탭으로 추가. 새 화면을 만들지 않는다.
  • 가져오기 클릭 → POST /api/ingest/url 단일 엔드포인트 (v2 FastAPI).
  • 서버 SSRF 가드 — http/https만, 사설 IP 대역 차단, 리다이렉트·응답 크기 제한.

구현 메모 — 화면 2

  • 이미지 탐색 3단 폴백 — og:image → JSON-LD(schema.org/Product) → 최대 이미지 휴리스틱. 사이트별 스크래퍼는 만들지 않는다.
  • 상품명 텍스트(og:title / JSON-LD name)는 기존 clothingMeta 이름 신호와 warmthDerivation typeLabel 규칙으로 흘려보낸다.
  • 단계 상태는 서버 응답 한 번으로 받아도 되고(단순), SSE로 흘려도 된다(선택). 1차는 단순 응답.

구현 메모 — 화면 3 (핵심)

  • "색상 분석 보기" → 지금 옷장에서 쓰는 ColorInsightModal을 그대로 연다. sourceLabel만 "URL 분석 완료 화면"으로 바뀐다 (이미 props로 지원).
  • 저장 시 스키마 — sourceType: 'url', sourceRef: 원본 URL, analysis.status: 'draft' → 'confirmed'. 전부 기존 필드.
  • 병합 경로는 하나 — 기존 수동 등록과 같은 저장 함수를 태운다. URL 전용 저장 로직을 만들지 않는다.
  • 이미지는 imageStore(IndexedDB)로 로컬 저장. 재배포 없음.

방법 2

사진 업로드로 전환

저장해 둔 상품 이미지를 직접 올려도 같은 분석을 거칩니다.

구현 메모 — 화면 4

  • 실패는 오류가 아니라 정상 시나리오다. 에러 코드가 아니라 다음 행동을 안내한다.
  • 이미지 주소 직접 입력은 화면 1과 같은 엔드포인트로 — Content-Type이 이미지면 파싱 단계를 건너뛴다.
  • 시연에서는 사전 검증한 URL 2~3개만 쓴다. 골든 패스는 네트워크 없이 완주된다.
구현 순서 제안 — ① 서버 프록시(+SSRF 가드) ② 화면 1·3 UI와 초안→확정 저장 ③ 파싱 3단 폴백 ④ 누끼·색 추출 연동 ⑤ 화면 2·4 상태 처리. 각 단계는 저장된 HTML 픽스처로 오프라인 테스트한다.
